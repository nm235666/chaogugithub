# 文档生命周期规则（2026-04-18）

## 1. 目标

确保文档“只保留当前可用知识”，并防止阶段性材料、测试产物、临时分析再次进入主仓库。

## 2. 分层规则

1. `Active`：对当前系统状态负责，必须持续维护。  
2. `Consolidated`：同主题唯一入口，减少重复文档。  
3. `Local Archive`：本地保留追溯，不纳入 Git/GitHub。  

## 3. 新文档进入条件

仅当满足以下之一，才允许进入 `docs/`：
- 定义当前系统真实行为（路由、接口、运行、数据结构、任务编排）。
- 作为唯一执行入口（当前计划、当前审计、当前规范）。
- 被 `DOCS_INDEX` 明确登记并指定 Owner。

不满足时，直接进入 `local_archive/docs/`。

## 4. 归档触发条件

出现以下任一条件即归档：
- 被新文档完全替代。
- 属于日期型阶段报告且超过当前执行窗口。
- 只保留历史留痕，不再作为“当前状态”承诺。
- 内容属于实验、提案、一次性分析输出。

## 5. 禁止提交项

以下内容禁止进入 Git：
- `local_archive/**`
- `apps/web/playwright-report/**`
- `apps/web/test-results/**`
- 运行与临时高噪音产物（`runtime/**`, `tmp/**`, `logs/**` 归档子路径）
- 外部同步数据目录（如 `external/strategy`）

## 6. 提交流程要求

1. 提交前必须检查 `docs/DOCS_INDEX.md` 是否同步。  
2. 变更涉及主链行为时，必须同步更新对应 Active 文档。  
3. 发现过期文档时，优先迁移到 `local_archive/docs/`，不保留在 `docs/` 根层。
