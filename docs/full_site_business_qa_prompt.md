四、测试执行策略
==================================================
采用分批交付，而不是一次性笼统扫站。

批次规则：
- 一次只处理一个可交付批次。
- 每批覆盖 2 到 4 个强相关模块，或一条完整业务链路。
- 每批完成后必须输出：
  - 本批测试了什么。
  - 实际结果。
  - 发现的问题。
  - 验证结果。
  - 剩余风险。
  - 下一批建议。

默认优先顺序：
1. 登录、退出、会话、首页、导航主链路。
2. 页面展示与数据加载。
3. 核心业务列表、搜索、筛选、排序、分页、详情。
4. 创建、编辑、删除、保存、取消、提交等关键操作。
5. 弹窗、表单、校验、提示反馈。
6. 权限差异、异常流程、边界情况。
7. 次要页面与补充功能。

==================================================
六、业务正确性评估
==================================================
本次测试不仅检查页面是否能打开、按钮是否能点击、数据是否加载，还必须评估每个业务功能“回答是否正确、结论是否可用、动作是否合理”。

核心要求：
- 每个业务模块都必须判断：页面给出的业务结论是否真正回答了用户问题。
- 不允许只因为页面有数据、有标签、有按钮，就判定通过。
- 如果页面展示的是泛化文案、占位答案、无关结论、旧数据、不可执行建议，应判定为“不通过”或“部分通过”。
- 业务功能验收优先于 UI 可用性：如果页面能正常展示、按钮能点击，但业务结论错误、不可执行、不可解释，仍判定该模块不通过。
- 业务结论必须满足：有依据、有对象、有动作、有时效、有回查路径。

业务评估维度：
1. 问题匹配度：页面回答的内容是否正面回答了标题或用户问题。
2. 结论明确度：是否给出明确判断，而不是模糊描述。
3. 对象完整度：涉及股票、主题、群聊、荐股人、订单、任务时，是否给出具体对象。
4. 证据充分度：是否展示来源、时间、样本量、理由或规则链路。
5. 动作可执行性：是否告诉用户下一步该做什么，按钮或跳转是否可达。
6. 时效正确性：是否使用当日或指定周期数据，是否误用历史旧数据。
7. 口径一致性：列表、详情、统计卡、六问、市场结论等页面是否说法一致。
8. 权限一致性：admin/pro/limited 下业务能力、限制和引导是否合理。
9. 空态合理性：无数据时是否说明缺什么、为什么缺、下一步去哪做。
10. 风险表达：涉及投资判断时，是否展示风险、置信度或需要人工复核提示。

业务正确性输出格式：

业务问题：
页面答案：
业务判断：正确 / 不正确 / 部分正确 / 需人工复核
判断依据：
缺失证据：
业务影响：
建议修复方向：

==================================================
七、重点业务功能专项
==================================================
对以下重点功能必须做业务正确性评估。

1. 今日六问
- Q1：是否真正回答“今天最该关注的短线主题是什么”。必须包含主题 + 代表标的，而不是只有新闻摘要或泛化概念。
- Q2：是否真正回答“关注池里哪些标的应新开仓”。必须排除 confirmed、rejected、deferred、executed、reviewed 等终态对象。
- Q3：是否回答“每只新开仓标的的建议仓位是多少”。必须包含现金比例、单票上限、风险预算等可执行数字。
- Q4：是否回答“持仓池里哪些标的应减仓/清仓”。没有持仓时空态是否合理。
- Q5：是否回答“卖出原因：止盈/止损/失效/风险切换”。必须使用今日或指定周期可追溯证据，不能用历史旧记录把今天判成已回答。
- Q6：是否回答“宏观状态是否变化，组合层该怎么应对”。必须包含变化判断 + 配置动作。
- 六问每一问都不能只检查“是否有文字”，必须检查“答案是否对应问题”。

2. 市场结论 / 今日交易主线
- 是否展示稳定的当日交易主线产物，而不是截断新闻 Markdown。
- 是否包含主线主题、代表标的、关键风险、证据来源和生成时间。
- “风险偏好、流动性、A股”等市场状态词不能被当作可交易主题。
- 市场页与六问 Q1 的结论是否一致。
- 如果置信度很低，页面是否提示需要人工复核，并说明缺少什么证据。

3. 主题热点 / 候选方向 / 股票短名单 / 候选漏斗
- 主题热点是否区分“可交易主题”和“市场状态主题”。
- 候选方向是否是可交易主题或方向，不应和股票短名单混淆。
- 股票短名单是否是经过筛选、可进入决策的具体标的。
- 候选漏斗状态是否合理推进，不能把终态对象重新当作新开仓候选。
- 候选标的是否有来源、信号强度、入选理由和下一步动作。

4. 决策看板 / 订单 / 动作链路
- 决策建议是否包含买入、卖出、观察、拒绝等明确动作。
- 每个动作是否有理由包、风险、证据、执行去向。
- 订单状态口径是否一致，例如 executed/done 是否兼容但不混乱。
- 从评分、短名单到决策、订单、复盘是否能形成闭环。
- Kill Switch、验证层等安全机制是否解释清楚，是否影响正常决策动作。

5. 宏观状态 / 配置建议
- 宏观状态是否区分短、中、长周期。
- 是否判断状态是否变化。
- 配置建议是否给出现金比例、单票上限、风险预算等数字。
- 宏观变化是否能映射到组合配置动作。
- 无变化时是否明确输出“维持策略”，而不是泛化空话。

6. 群聊投资信号 / 群聊详情 / 荐股人详情
- 群聊股票识别是否准确，ST 改名、别名、简称是否能匹配。
- 方向校验是否覆盖所有投资群，而不是只覆盖部分群。
- 命中、未命中、待校验的判定是否基于下个交易日或指定规则。
- 群聊详情是否能解释该群的样本量、准确率、代表股票、历史记录。
- 荐股人详情是否能解释该荐股人的样本量、准确率、所在群、历史记录。
- 荐股人准确率排序是否符合业务预期，不能低可信低准确率排第一。
- “无样本”是否真实无样本，还是因为过滤、名称匹配或校验任务未跑。

7. 数据源监控 / 任务调度 / 告警中心
- 数据源“暂无数据”是否真实无数据，还是接口或任务失败。
- 未恢复告警是否只保留仍失败的告警，已恢复服务是否自动清理。
- 失败任务是否展示可行动失败原因，而不是只有失败状态。
- “复制排障命令”是否可用，失败时是否有手动命令兜底。
- news_daily_summary_refresh、daily_postclose_update 等关键任务失败时，页面是否给出可定位原因。

8. 用户与会话管理 / 权限管理
- 用户列表是否能正常加载。
- 角色变更、启禁用、重置密码、额度调整等关键操作是否可点击、可提交、可反馈。
- 权限变更后页面能力是否真实变化。
- 会话强制下线是否有效。
- 操作是否有审计日志。

==================================================
八、问题发现加严规则
==================================================
必须额外执行以下专项，如环境允许：

1. 慢加载专项：高延迟下是否空白、错位、重复请求、状态错乱。
2. 并发/重复提交专项：双击、连击、回退重提是否幂等。
3. 权限越权专项：低权限访问高权限入口是否被拦截。
4. 数据一致性专项：列表、详情、统计数字是否一致。
5. 异常恢复专项：接口 4xx、5xx、超时后页面是否可恢复。
6. 文案与可理解性专项：提示是否可行动，是否告诉用户下一步。
7. 跨角色回归专项：同一功能在 admin/pro/limited 下是否一致合理。
8. 会话与安全专项：过期会话、退出后返回页、多标签页状态。
9. 视觉与响应式专项：桌面和移动端是否布局可读、关键操作可达。
10. 控制台与网络专项：记录控制台错误、失败请求、异常状态码。

以下情况必须记为 Bug：
- 页面展示了答案，但没有回答业务问题。
- 页面展示了“已回答/ready”，但答案不可执行。
- 页面使用旧数据、错周期数据或无关数据得出结论。
- 页面统计数字与列表/详情不一致。
- 页面主结论与其他页面同一功能口径不一致。
- 页面把状态、风险、新闻摘要误当成业务结论。
- 页面无法解释为什么给出某个建议。
- 页面没有给出下一步动作或回查证据。
- 页面按钮可见但不可点击，或点击后无反馈。
- 页面接口失败但无可行动错误提示。
